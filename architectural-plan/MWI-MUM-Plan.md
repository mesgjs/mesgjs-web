# MWI Mount/Unmount Monitor (MWIMUM) Architectural Plan

This document outlines the final hybrid architecture for the `MWIMUM`, which is responsible for managing component lifecycle events.

## 1. Guiding Principles

The `MWIMUM` architecture balances three core requirements:
1.  **Performance:** Initial page hydration must be fast and parallelizable.
2.  **Correctness:** Runtime DOM manipulations must be handled serially for a given element to prevent race conditions.
3.  **Developer Experience:** The system should default to a simple "convention over configuration" model while providing an escape hatch for advanced use cases.

## 2. The Hybrid Subscription Model

The `MWIMUM` is initially driven by a static configuration map generated by the Server-Side Renderer (SSR). This map connects element IDs to their corresponding lifecycle behaviors.

### 2.1. Convention-Based Subscription (Default)

For most components, the subscription data is a simple **string** specifying the `PascalCase` or `camelCase` name of the Mesgjs interface for the Component Handler.

*   **Example:** `{ "my-element-id": "MyComponentHandler" }`
*   **Behavior:** By convention, the `MWIMUM` sends a `(mount)` message when the element appears and an `(unmount)` message when it disappears.

### 2.2. Configuration-Based Subscription (Override)

For components needing custom lifecycle logic, the subscription data is an **object** that provides an `interface` and can optionally override the default mount and/or unmount messages.

*   **Structure:**
    *   `interface`: (Required) The string name of the Component Handler's Mesgjs interface.
    *   `mountMessage`: (Optional) The message to send on mount. If absent, `['mount']` is used.
    *   `unmountMessage`: (Optional) The message to send on unmount. If absent, `['unmount']` is used.

*   **Example:**
    ```javascript
    {
      "my-datepicker-id": {
        interface: "DatePicker",
        mountMessage: ["init", { range: "2025" }],
        unmountMessage: ["cleanup", { saveState: true }]
      }
    }
    ```
*   **Behavior:** The `MWIMUM` sends the precisely defined messages for each event. If an override message is not provided for an event, the convention-based default (`mount`/`unmount`) is used.

## 3. Programmatic Registration (CSR)

For purely client-side rendered VNodes not in the SSR payload, a programmatic method is used.

*   `mum.register(element, vnodeReceiver)`: A VNode registers its DOM `element` and its Mesgjs receiver function.
*   **Behavior:** When the `element` is removed, the `MWIMUM` sends a conventional `(unmount)` message directly to the registered `vnodeReceiver`.

## 4. Processing Architecture

The processing logic remains the same, leveraging parallelism for `init()` and serial execution for runtime `MutationObserver` updates to ensure both performance and correctness. The handler logic will now simply check for `mountMessage` or `unmountMessage` on the subscription object before falling back to the default convention.
