# Architectural Decision Log (ADL)

This document serves as an immutable log of architecturally significant decisions made during the project. It is intended to provide context for future development and to record the reasoning behind key choices.

Entries should be added for any decision that has a non-trivial impact on the project's architecture, maintainability, or security. This is the designated location for recording the outcomes of "Intelligent Disobedience" discussions.

---

## 2025-07-29: Hardening of h.* Primitives and Core Handlers

-   **Context:** Following a comprehensive architectural audit, several gaps and potential security weaknesses were identified in the planned implementation of the core `h.*` primitive components.
-   **Discussion:** The primary issue was the implicit behavior of the default `h.*` handler, which could render any tag except those with dedicated handlers (like `script` and `style`). This "deny-list" approach is less secure than an "allow-list." Additionally, core functionalities for fragments, text nodes, comments, and titles were not explicitly defined.
-   **Decision:**
    1.  The standard `h.*` handler **MUST** operate on a strict "allow-list" basis. It will only render a pre-configured, enumerated set of safe HTML tags. It is not responsible for any tag that has a dedicated handler.
    2.  New declarative handlers **MUST** be created for `h.FRAG` (fragment), `h.TEXT` (text node), and `h.COMMENT` (HTML comment).
    3.  A new specialized handler **MUST** be created for `h.title`.
    4.  The `h.title`, `h.TEXT`, and `h.COMMENT` components should accept their content via a `text` attribute, as they do not contain child nodes. The `h.title` and `h.TEXT` handlers' CSR implementations should support a reactive value for this attribute.
-   **Reasoning:** Switching to an "allow-list" model for the `h.*` primitives significantly improves security by default. Explicitly defining handlers for non-element nodes like fragments and comments clarifies the architecture and makes the system more robust. The specialized `h.title` handler is necessary for its unique role in the document `<head>`.

## 2025-08-04: V4 Architectural Refinements

-   **Context:** During initial planning for the V4 core handlers, a review of the V4 architecture (`MWI-V4-Core-Architecture.md`) and subsequent discussion clarified several implementation details.
-   **Discussion:** The V4 architecture uses a lightweight, serializable `MWISSRInfo` object instead of a complex VNode class. This simplification prompted a review of how text and other special nodes should be represented. The initial plan from 2025-07-29 to use a `text` attribute was also revisited.
-   **Decision:**
    1.  The `MWISSRInfo` object generated by the `h.TEXT` component handler **MUST** have a `type` of `'h.TEXT'`, not the more generic `'#text'`, to maintain a one-to-one correspondence between component and info types.
    2.  For components that accept simple text content but do not render HTML children (e.g., `h.COMMENT`, `h.title`, `h.doctype`), the standard attribute **MUST** be `content`. This supersedes the previous decision to use `text`.
    3.  The `h.doctype` component is a simple primitive and is not part of the `m.doc` orchestration. Its handler will accept a `content` attribute, escape the value, and render it inside the final doctype string.
-   **Reasoning:** Aligning the info `type` with the component name (`h.TEXT`) makes the system more transparent and easier to debug. Standardizing on a `content` attribute for all text-only components creates a more consistent and predictable component API. Clarifying `h.doctype`'s role simplifies its implementation and removes an unnecessary dependency on the more complex `m.doc` orchestrator.

## 2025-08-05: Component Registration and Capability Model Refinement

-   **Context:** The initial MWI V4 architecture (`MWI-V4-Core-Architecture.md`) defined a rendering pipeline that transformed component data into a serializable `MWISSRInfo` tree, but it crucially did not specify *what* logic was responsible for this transformation. This left a significant architectural gap: how does a declarative component structure (e.g., `[my-component]`) get converted into a tree of primitive `h.*` components?
-   **Discussion:** The V3 architecture used complex, payload-returning component handlers, a model the V4 architecture explicitly replaces. A new model was needed. The answer is the "HTML Component Generator," a function that encapsulates the logic for transforming a high-level component into its primitive representation. This led to a discussion about how to register these generators securely. The "push" model—whereby modules proactively register their generators with a central registry—was determined to be simpler and more secure than a "pull" model where the registry discovers them.
-   **Decision:**
    1.  The concept of an "HTML Component Generator" is formally adopted as the mechanism for transforming a semantic component's data into an `MWISSRInfo` tree. This is a core part of the V4 architecture.
    2.  The `MWIComponentRegistry` **MUST** expose a public Mesgjs interface for component generator registration. The previously considered "private registration handshake" and `HTMLGeneratorService` concepts are deprecated.
    3.  The registry's handler **MUST** enforce that the calling module has the `"mwi.htmlGenerator"` capability in its `modcaps` metadata. This is the primary security gate for this privileged operation.
    4.  The `mwi.htmlGenerator.`-prefixed feature promise (`featpro`) is no longer required for generator modules.
    5.  The module metadata key for declaring capabilities is officially `modcaps`. All documentation must be updated to reflect this.
    6.  The feature promise for the component registry is officially `mwi.componentRegistryReady`. All documentation must be updated to correct the erroneous `mwi.registry.ready` value.
-   **Reasoning:** This set of decisions formally closes a major gap in the V4 architecture by defining the generator model. It simplifies the component registration process, making it more direct while placing security enforcement squarely on the `MWIComponentRegistry`. Aligning all documentation with the correct `modcaps` and feature promise names eliminates ambiguity and prevents future implementation errors.
